# Week 2-3 - 半独立到独立执行与发布

第二、三周是从"在引导下完成一个工单"过渡到"独立交付并发布"的关键阶段。每周有明确的目标、协作方式与产出标准，由 mentor 评审进度并记录到 Wiki。完整的入职节奏与评估方式见 [[wps-payroll-qa-onboarding-plan]]，前置阶段见 [[wps-payroll-qa-onboarding-week1]]。

## Week 2 – Lead a Ticket (Semi-Independent)

### 目标
- Drive a ticket end-to-end
- 团队仅在被询问时支援，**不主动提示下一步**

### Key Learning Points
- 独立读懂需求并设计/编写测试用例
- 在 Zephyr 中链接用例、执行并独立提 bug
- 练习判断**何时**以及**向谁**提问：
  - 环境问题
  - 需求不清
  - 合作方/集成相关问题（YSE via PPC、CBUAE、TradeII 等）

### 协作方式
- 团队指派一个由你**主导**的工单
- 卡住时你来问，团队回答但不接管
- 团队进行**比 Week 1 更轻量的 review**：关注结果与判断逻辑，而非每一步细节

### Expected Outcome
- 在最少干预下完成一个工单
- 测试用例与执行能独立成立，仅需轻微修正

## Week 3 – Independent Execution & Release

### 目标
- 独立完成一个工单并推动到发布上线

### Key Learning Points
- 完全负责一个工单：requirement → test cases → execution → bug reporting → sign-off → **release / go-live**
- 主动与 Developers、PM、stakeholders 沟通
- 管理测试进度与问题跟进

### 协作方式
- 工作内容与决策都由你负责
- 团队仅在发布前提供 **final review / sign-off**
- 发布后做 debrief：
  - 哪些做得好
  - 哪些需要改进
  - 下一步可深入的方向（如 PAF-type companies、central-bank file flows、partner integration testing）

### Expected Outcome
- 独立完成并交付一个工单，达到团队标准
- 测试结论清晰、可靠、文档完备

## 通过标准回顾

通过 onboarding 意味着能够独立支持 WPS Payroll QA 工作，并将一个工单带到 release。评估结果分为：

- ✅ **Pass** – Ready to move forward
- ⚠️ **High Risk** – Improvement required
- ❌ **Fail** – Onboarding not passed
